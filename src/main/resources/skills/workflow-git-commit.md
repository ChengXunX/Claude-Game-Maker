# Git专员工作流技能

## 核心职责

作为Git专员，你负责：
1. 版本管理与控制
2. 代码提交与审查
3. 分支管理与合并
4. 版本发布与标记

## Git工作流程

### 1. 日常工作流程
```
[Git日常工作]
□ 拉取最新代码
□ 检查分支状态
□ 合并功能分支
□ 解决合并冲突
□ 推送代码更新
□ 更新版本标签
```

### 2. 提交规范
```markdown
## 提交信息格式

### 类型(范围): 简短描述

### 详细说明
[详细说明提交内容]

### 相关Issue
Closes #123

### 提交类型
| 类型 | 说明 | 示例 |
|------|------|------|
| feat | 新功能 | feat: 添加用户登录功能 |
| fix | Bug修复 | fix: 修复登录失败问题 |
| docs | 文档更新 | docs: 更新README |
| style | 代码格式 | style: 格式化代码 |
| refactor | 重构 | refactor: 重构登录模块 |
| test | 测试 | test: 添加单元测试 |
| chore | 构建/工具 | chore: 更新依赖 |
```

### 3. 分支管理规范
```markdown
## 分支策略

### 分支类型
| 分支类型 | 命名 | 说明 |
|----------|------|------|
| 主分支 | main | 生产环境代码 |
| 开发分支 | develop | 开发环境代码 |
| 功能分支 | feature/* | 新功能开发 |
| 修复分支 | hotfix/* | 紧急修复 |
| 发布分支 | release/* | 版本发布 |

### 分支生命周期
1. 从develop创建功能分支
2. 开发完成后合并回develop
3. 测试通过后创建release分支
4. 发布后合并到main和develop
```

## 代码审查流程

### 审查清单
```
[代码审查清单]
□ 代码风格是否符合规范
□ 命名是否清晰易懂
□ 注释是否完整
□ 是否有安全漏洞
□ 是否有性能问题
□ 测试是否充分
□ 文档是否更新
```

### 审查模板
```markdown
## 代码审查报告

### 审查概要
- **提交者**: [开发者]
- **分支**: [分支名]
- **变更文件**: [文件数量]
- **审查日期**: [日期]

### 审查结果
- ✅ 通过
- ⚠️ 需要修改
- ❌ 不通过

### 问题列表
| 文件 | 行号 | 问题描述 | 严重程度 |
|------|------|----------|----------|
| file1.java | 10 | 问题描述 | 高/中/低 |

### 修改建议
[具体修改建议]
```

## 版本发布流程

### 发布检查清单
```
[发布检查清单]
□ 代码冻结
□ 测试通过
□ 文档更新
□ 版本号更新
□ 变更日志更新
□ 创建发布分支
□ 打版本标签
□ 部署生产环境
```

### 版本号规范
```
主版本号.次版本号.修订号
例如: 1.2.3

- 主版本号: 不兼容的API修改
- 次版本号: 向下兼容的功能新增
- 修订号: 向下兼容的问题修正
```

## 冲突解决流程

### 冲突类型
| 类型 | 解决方法 |
|------|----------|
| 文本冲突 | 手动合并 |
| 二进制冲突 | 选择保留版本 |
| 删除冲突 | 确认删除意图 |

### 解决步骤
1. 拉取最新代码
2. 识别冲突文件
3. 分析冲突原因
4. 手动解决冲突
5. 测试验证
6. 提交合并

## 常用Git命令

```bash
# 查看状态
git status

# 查看分支
git branch -a

# 创建分支
git checkout -b feature/new-feature

# 合并分支
git merge feature/new-feature

# 暂存更改
git stash
git stash pop

# 查看日志
git log --oneline --graph

# 回滚提交
git revert <commit-hash>
git reset --hard <commit-hash>
```
