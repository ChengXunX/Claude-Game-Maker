# 代码审查技能

## 审查原则

### 核心理念
- **建设性**: 提出改进建议而非批评
- **具体性**: 指出具体问题和解决方案
- **学习性**: 相互学习共同成长
- **效率性**: 及时审查不阻塞开发

## 游戏代码审查要点

### 1. 游戏逻辑审查
```
审查要点：
- 游戏规则是否正确实现
- 状态管理是否清晰
- 碰撞检测是否准确
- 物理模拟是否真实

常见问题：
- 状态混乱：使用状态机管理
- 碰撞穿透：检查碰撞体设置
- 物理异常：检查物理参数
- 逻辑错误：仔细测试边界
```

### 2. 游戏性能审查
```
审查要点：
- 对象池使用
- 内存管理
- 渲染优化
- 网络优化

常见问题：
- 内存泄漏：对象未销毁
- 帧率下降：渲染过多对象
- 加载缓慢：资源未优化
- 网络延迟：包太大
```

### 3. 游戏UI审查
```
审查要点：
- 层级管理
- 响应式适配
- 动画流畅度
- 交互体验

常见问题：
- 层级混乱：统一层级规范
- 适配问题：测试不同设备
- 动画卡顿：优化动画性能
- 交互不便：简化操作流程
```

## 审查意见模板

### 问题描述
```markdown
## 问题描述

**文件**: src/game/BattleSystem.js
**行号**: 45-60
**类型**: 性能问题

**问题**: 
在update函数中每帧都创建新对象，会导致GC压力。

**建议**:
使用对象池或预分配对象。

**示例**:
```javascript
// 修改前
update() {
  const effect = new Effect(); // 每帧创建新对象
}

// 修改后
update() {
  const effect = this.effectPool.get(); // 从对象池获取
}
```
```

### 优点表扬
```markdown
## 优点表扬

**文件**: src/game/Player.js
**行号**: 100-120
**类型**: 代码质量

**优点**:
状态机实现清晰，易于理解和维护。

**说明**:
使用状态机管理玩家状态，每个状态有明确的enter/update/exit方法。
```

## 审查流程

### 1. 提交审查
```
提交信息格式：
- 标题：简短描述变更
- 描述：详细说明变更内容
- 关联：关联issue或需求
- 测试：说明测试情况
```

### 2. 审查反馈
```
反馈类型：
- 必须修复：阻塞性问题
- 建议修复：非阻塞性问题
- 可选优化：优化建议
- 问题澄清：需要解释
```

### 3. 修复验证
```
验证步骤：
1. 修复问题
2. 运行测试
3. 提交修复
4. 请求重新审查
```

## 常见错误

1. **审查不及时**：要尽快审查，不阻塞开发
2. **审查不仔细**：要仔细检查每个细节
3. **反馈不具体**：要指出具体问题和解决方案
4. **只找问题不表扬**：要肯定好的代码
5. **不跟进修复**：要确认问题已修复
6. **迭代审查不充分**：要充分审查迭代代码
7. **不参与评审**：要参与评审验证质量
8. **不总结经验**：要及时总结审查经验
