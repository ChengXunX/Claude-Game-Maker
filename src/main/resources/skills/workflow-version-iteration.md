# 版本迭代工作流

## 核心理念

**迭代式开发**：每个迭代周期产出一个可玩的版本，基于反馈持续改进。

## 迭代生命周期

```
┌─────────────────────────────────────────────────────────────┐
│                      版本迭代生命周期                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐  │
│  │ 规划 │───▶│ 开发 │───▶│ 测试 │───▶│ 评审 │───▶│ 回顾 │  │
│  └──────┘    └──────┘    └──────┘    └──────┘    └──────┘  │
│     │           │           │           │           │       │
│     ▼           ▼           ▼           ▼           ▼       │
│  确定目标    实现功能    验证质量    评估成果    总结经验    │
│  分配任务    编写代码    修复bug     决策下一步  知识沉淀    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 第一阶段：规划（1-2天）

### 1.1 制作人：制定迭代计划
```
[迭代规划清单]
□ 回顾上一迭代的评审结果
□ 确定本次迭代目标（1-3个核心目标）
□ 分解任务并估算工作量
□ 分配任务给对应Agent
□ 确定验收标准
□ 识别风险和依赖
```

### 1.2 迭代计划模板
```markdown
# 迭代 [版本号] 计划

## 迭代目标
- 目标1：[具体、可测量的目标]
- 目标2：[具体、可测量的目标]

## 任务清单
| 任务 | 负责人 | 预计工时 | 优先级 | 依赖 |
|------|--------|----------|--------|------|
| 任务1 | Agent A | 2天 | P0 | 无 |
| 任务2 | Agent B | 3天 | P0 | 任务1 |
| 任务3 | Agent C | 1天 | P1 | 无 |

## 验收标准
- [ ] 标准1：[具体、可验证的标准]
- [ ] 标准2：[具体、可验证的标准]

## 风险识别
| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 风险1 | 高 | 中 | 措施1 |

## 时间节点
- 规划完成：[日期]
- 开发完成：[日期]
- 测试完成：[日期]
- 评审日期：[日期]
```

## 迭代节奏

### 短迭代（1周）
```
周一：规划（0.5天）+ 开发（0.5天）
周二-周三：开发（2天）
周四：测试（1天）
周五：评审（0.5天）+ 回顾（0.5天）
```

### 标准迭代（2周）
```
第1天：规划
第2-7天：开发
第8-9天：测试
第10天：评审 + 回顾
```

### 长迭代（3-4周）
```
第1-2天：规划
第3-15天：开发
第16-18天：测试
第19-20天：评审 + 回顾
```

## 常见错误

1. **规划不充分**：要充分规划，明确目标和任务
2. **任务分配不合理**：要根据Agent能力分配
3. **进度跟踪不及时**：要每日检查进度
4. **测试不充分**：要全面测试
5. **评审走过场**：要认真评审，收集反馈
6. **回顾不深入**：要深入总结，沉淀知识
7. **知识不沉淀**：要及时更新知识库
